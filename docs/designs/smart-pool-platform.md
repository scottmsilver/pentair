## Vision

### 10x Check
Current plan: Rust CLI for pool control from a terminal. 15% of the vision.

10x version: A personal smart pool platform. Voice control via Google Home ("Hey Google, warm up the spa to 104"), native Android app with real-time dashboard and one-tap controls, conversational AI for natural language pool management, push notifications for freeze alerts and spa-ready events. The protocol library serves as the foundation for CLI, daemon, app, and voice integrations.

### Platonic Ideal
You walk outside. Glance at your phone widget: pool 82°F, spa off, pump running. Say "Hey Google, warm up the spa to 104." It responds: "Heating spa to 104. Should be ready in about 20 minutes. Want me to turn on the lights?" You say yes.

20 minutes later, your watch buzzes: "Spa is ready!"

On vacation, push notification: "Pool chemistry alert — pH is 7.8, above your 7.6 target." Open the app, see the week's pH trend.

Your pool is a first-class smart home device, just like lights or thermostat.

### Crate structure (Rust workspace)

```
pentair-protocol/    Wire protocol: types, encode, decode (no IO)
pentair-client/      Async TCP/UDP client (tokio)
pentair-daemon/      Long-running service: REST API + WebSocket + state cache
pentair-cli/         Command-line tool (talks to daemon or direct to adapter)
```

### Key decisions
- **API protocol**: REST + WebSocket (REST for commands, WebSocket for real-time push)
- **Network reach**: LAN-only (WiFi). Cloud relay deferred. VPN for remote access.
- **Authentication**: None (matches ScreenLogic adapter's trust model — LAN = trusted)
- **Android app**: Kotlin + Jetpack Compose, talks to daemon's REST API
- **Google Home**: Local fulfillment via Smart Home API device traits (OnOff, TemperatureSetting, Modes)
- **LLM layer**: Claude/Gemini with tool-use calling daemon's REST API

## Phasing

**Phase 1: Protocol + CLI + Daemon**
- pentair-protocol: wire types, encode/decode, all message parsers
- pentair-client: discovery, connection, login, all commands
- pentair-daemon: persistent connection, state cache, REST API, WebSocket push
- pentair-cli: all commands, talks to daemon or direct

**Phase 2: Android App + Google Home**
- Android app: dashboard, circuit control, heat control, lights, schedules, chemistry
- Google Home: Smart Home fulfillment (OnOff, TemperatureSetting, Modes)
- Push notifications via FCM

**Phase 3: Intelligence**
- Conversational AI layer (LLM tool-use)
- Chat interface in Android app
- Heat-up time estimates
- Sunset-aware scheduling
- Friendly circuit names (config-driven)

## Deferred
- Pool party macros
- Remote access / cloud relay
- Temperature history + analytics
- Energy cost estimates
- iOS app
- Web UI
- Home Assistant integration (existing Python library handles this)
- Multi-controller support
